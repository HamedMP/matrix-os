# Spec 051: Framework Apps (Build-to-Static)

**Goal**: Enable the kernel agent to scaffold, build, and serve full React/Vite apps as static bundles inside the existing iframe sandbox. Apps use the same `MatrixOS.db.*` data layer (spec 050), the same OS bridge for theming and IPC, and the same `/files/` serving path. The agent runs the build; users can ask for a rebuild anytime.

**Depends on**: 050-app-data-layer (Postgres backend, `MatrixOS.db.*` bridge client)

## Problem

1. Apps are single-file HTML with inline CSS and vanilla JS -- no component model, no npm packages, no framework
2. The agent cannot use React, Tailwind, Chart.js, Three.js, or any library that requires a build step
3. Complex apps (dashboards, multi-view tools, data visualizations) are impractical as a single HTML file
4. No typed SDK -- iframe apps rely on an injected `window.MatrixOS` global with no autocompletion or documentation
5. System-level tools (ffmpeg, imagemagick, wkhtmltopdf) are unavailable for apps that need them

## Design Rationale

The constitution prefers CDN imports for simple generated HTML apps. Framework apps are the escalation path for when single-file HTML with CDN imports is insufficient: multi-view apps, complex state management, typed SDK, npm ecosystem access. Single-file HTML remains the default for simple tools (calculator, clock, single-screen utilities). The agent decides which approach to use based on complexity.

## Solution

### Architecture

```
User: "Build me a project tracker with React"
  |
  v
1. SCAFFOLD -- Agent writes ~/apps/{slug}/
     matrix.json, package.json, vite.config.ts,
     index.html, src/App.tsx, src/main.tsx
  |
  v
2. BUILD -- Agent runs in ~/apps/{slug}/:
     sudo apk add --no-cache ...   (if system deps declared)
     pnpm install                   (npm deps)
     pnpm build                     (vite build -> dist/)
  |
  v
3. SERVE -- Gateway serves dist/ via existing /files/ route
     iframe src = /files/apps/{slug}/dist/index.html
  |
  v
4. REBUILD -- User asks "rebuild my project tracker"
     Agent modifies source if needed, re-runs step 2
```

No new gateway routes. No process manager. The `dist/` folder is static files under `/files/apps/`. The iframe sandbox and OS bridge work unchanged.

### A: Manifest Extensions

Extend the existing `AppManifestSchema` in `app-manifest.ts`. The `entry` and `framework` fields already exist as optional strings. Add `system` and `build`:

```ts
// Additions to AppManifestSchema in app-manifest.ts
system: z.array(z.string()).default([]),
build: z.object({
  install: z.string().default("pnpm install"),
  command: z.string().default("pnpm build"),
  output: z.string().default("dist"),
}).optional(),
```

Example manifest:

```json
{
  "name": "Project Tracker",
  "runtime": "node",
  "framework": "vite-react",
  "entry": "dist/index.html",
  "system": ["ffmpeg"],
  "build": {
    "install": "pnpm install",
    "command": "pnpm build",
    "output": "dist"
  },
  "storage": {
    "tables": {
      "projects": {
        "columns": { "name": "text", "status": "text", "due": "timestamptz" },
        "indexes": ["status"]
      }
    }
  }
}
```

New fields:

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `system` | `string[]` | `[]` | Alpine packages to install via `sudo apk add` |
| `build.install` | `string` | `"pnpm install"` | Dependency install command |
| `build.command` | `string` | `"pnpm build"` | Build command |
| `build.output` | `string` | `"dist"` | Output directory relative to app root |

Existing fields used by this spec (no changes needed):

| Field | Description |
|-------|-------------|
| `entry` | Entry file path, e.g. `"dist/index.html"` (already optional string) |
| `framework` | Framework identifier, e.g. `"vite-react"` (already optional string) |
| `runtime` | Already supports `"node"` |

Static apps without `build` or `runtime: "static"` continue to work unchanged.

### B: Vite Config and Base Path

Framework apps must set the correct `base` so assets resolve through the gateway file server:

```ts
// vite.config.ts (generated by agent)
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';

export default defineConfig({
  plugins: [react()],
  base: '/files/apps/project-tracker/dist/',
  build: { outDir: 'dist' },
});
```

The `base` matches the gateway's `/files/{path}` route. All asset URLs resolve correctly through the existing static file server without any gateway changes.

### C: MatrixOS SDK Package

A typed npm package that framework apps import instead of relying on the injected `window.MatrixOS` global:

```
packages/sdk/             (workspace package: @matrix-os/sdk)
  src/
    core.ts               MatrixOS client (db via fetch, IPC via postMessage)
    react.ts              React hooks (useMatrixDB, useTheme)
    types.ts              Shared types (ThemeVars, DBQuery, etc.)
    index.ts              Main export
  package.json
  tsconfig.json
```

**Distribution**: `@matrix-os/sdk` is a pnpm workspace package in `packages/sdk/`. Framework apps resolve it via a `file:` dependency in their `package.json`:

```json
{
  "dependencies": {
    "@matrix-os/sdk": "file:/app/packages/sdk"
  }
}
```

In Docker, the monorepo is mounted at `/app`, so `/app/packages/sdk` is always available. For future npm publishing, the `file:` path is swapped for a version number -- no code changes needed.

**Core client** (`core.ts`):

The core client provides two transport layers depending on the operation:

- `db.*` -- always uses HTTP fetch to `/api/bridge/query` (same as today's `MatrixOS.db.*`)
- `generate()`, `openApp()`, `navigate()` -- uses `postMessage` to parent frame (iframe-only operations)
- `getTheme()` -- reads CSS custom properties from the document

```ts
import { createMatrixOS } from '@matrix-os/sdk';

const os = createMatrixOS({ app: 'project-tracker' });

// Data access -- HTTP fetch to /api/bridge/query
const tasks = await os.db.find('tasks', { where: { done: false } });
await os.db.insert('tasks', { name: 'Ship v1', status: 'todo' });

// Theme -- reads CSS custom properties
const theme = os.getTheme();

// IPC -- postMessage to parent frame (iframe-only)
os.generate('Add a burndown chart to this app');
os.openApp('calendar', '/files/apps/calendar/index.html');
```

**React hooks** (`react.ts`):

```tsx
import { useMatrixDB, useTheme } from '@matrix-os/sdk/react';

function TaskList() {
  const { data: tasks, loading, refetch } = useMatrixDB('tasks', {
    where: { done: false },
    orderBy: { due: 'asc' },
  });

  const theme = useTheme();

  if (loading) return <div>Loading...</div>;
  return tasks.map(t => <TaskRow key={t.id} task={t} />);
}
```

**Reactivity model**: `useMatrixDB` fetches data on mount and subscribes to `os:data-change` postMessage events (same mechanism as today's `MatrixOS.db.onChange`). When a `data:change` event arrives for the relevant table, the hook re-fetches. No polling, no WebSocket -- reuses the existing event flow through `AppViewer`.

**Theme integration**: `useTheme()` reads CSS custom properties on mount and listens for `os:theme-update` postMessage events. On theme change, the hook updates state (triggering React re-render) and patches CSS custom properties on `:root` so Tailwind `theme()` and plain CSS both work.

### D: Build Flow

The agent executes builds using its existing shell/Bash tools. No new IPC tools required.

**Step 1: System dependencies** (if `manifest.system` is non-empty):
```bash
cd ~/apps/{slug}
sudo apk add --no-cache ffmpeg graphicsmagick
```

Security note: `sudo apk` is acceptable because each user has their own container -- the isolation boundary is the container, not the process. A user can only install packages in their own environment.

**Step 2: npm dependencies**:
```bash
cd ~/apps/{slug}
pnpm install
```

**Step 3: Build**:
```bash
cd ~/apps/{slug}
pnpm build
```

**Step 4: Verify**:
Agent checks that `{build.output}/index.html` exists (default: `dist/index.html`). If the build fails, the agent reads the error output, fixes the source, and retries.

**Rebuild**: User says "rebuild my project tracker" or "update the tracker". The agent modifies source files if needed, then re-runs steps 1-4. The gateway serves the new `dist/` immediately (no cache headers on dev, ETag on production).

### E: Gateway and Shell Changes

**Gateway (`listApps` / `apps.ts`)**:

When scanning directory apps, read `manifest.entry` and use it instead of the hardcoded `"index.html"`:

```ts
// In scanAppsDir, when a manifest is found for a directory app:
const entry = manifest.entry ?? "index.html";
result.push({
  name: manifest.name,
  // ...other fields
  file: `${relativePath}/${entry}`,
  path: `/files/apps/${relativePath}/${entry}`,
});
```

**Gateway (`listApps` visibility)**: Only list framework apps that have been built. If `manifest.build` is defined but `{appDir}/{build.output}/index.html` does not exist, skip the app. This prevents unbuilt apps from appearing in the dock or Mission Control.

**AppManager (`app-manager.ts`)**:
- `register()` reads the `entry` field and sets `path` accordingly
- Status: `"running"` if the entry file exists, `"stopped"` if `manifest.build` is defined but output missing

**Shell (`AppViewer.tsx`)**:
- No changes needed. `AppViewer` already receives `path` from the window manager and renders an iframe at that path
- The OS bridge injection works identically -- `dist/index.html` is served from the same gateway origin

**Shell (`Desktop.tsx` / window manager)**:
- `openWindow()` already uses the `path` field from `listApps()` -- no changes needed

### F: Dockerfile Changes

Add to `Dockerfile.dev`:

```dockerfile
# Sudo for scoped package installation (apps with system deps)
RUN apk add --no-cache sudo \
 && echo "matrixos ALL=(root) NOPASSWD: /sbin/apk" >> /etc/sudoers.d/matrixos-apk \
 && chmod 0440 /etc/sudoers.d/matrixos-apk
```

Everything else (Node.js 24, pnpm, git, python3, make, g++) is already present.

### G: Agent Knowledge File

A new knowledge file at `~/agents/knowledge/framework-apps.md` teaches the kernel:

1. **When to use framework apps vs single-file HTML**: Single-file for simple tools (calculator, clock, single-screen utilities). Framework app for multi-view tools, apps needing npm packages, complex state management, or data-heavy UIs.

2. **Scaffold template**: Standard file structure for a Vite + React app:
   ```
   apps/{slug}/
     matrix.json
     package.json
     index.html          (Vite entry point)
     vite.config.ts
     src/
       main.tsx           (ReactDOM.createRoot)
       App.tsx            (root component)
       components/        (UI components)
     tsconfig.json
   ```

3. **SDK usage patterns**: Import from `@matrix-os/sdk/react`, use `useMatrixDB` for data, `useTheme` for theming, declare tables in `matrix.json` storage field.

4. **SDK resolution**: Add `"@matrix-os/sdk": "file:/app/packages/sdk"` to the app's `package.json` dependencies.

5. **Build commands**: `pnpm install && pnpm build`, verify `dist/index.html` exists.

6. **System dependencies**: Declare in `manifest.system`, install via `sudo apk add --no-cache`.

7. **Vite base path**: Must match `/files/apps/{slug}/dist/` for assets to resolve.

8. **Rebuild**: Modify source, run build again. User can ask "rebuild {app}" at any time.

### H: Example End-to-End

User: "Build me a habit tracker with React. I want to track daily habits, see streaks, and get a weekly summary chart."

Agent:
1. Creates `~/apps/habit-tracker/matrix.json`:
   ```json
   {
     "name": "Habit Tracker",
     "runtime": "node",
     "framework": "vite-react",
     "entry": "dist/index.html",
     "build": { "output": "dist" },
     "category": "productivity",
     "storage": {
       "tables": {
         "habits": {
           "columns": { "name": "text", "color": "text", "target_days": "integer" }
         },
         "entries": {
           "columns": { "habit_id": "uuid", "date": "date", "completed": "boolean" },
           "indexes": ["habit_id", "date"]
         }
       }
     }
   }
   ```

2. Creates `package.json` with `react`, `vite`, `@vitejs/plugin-react`, `@matrix-os/sdk` (`file:/app/packages/sdk`), `recharts`, `tailwindcss`

3. Creates `vite.config.ts` with `base: '/files/apps/habit-tracker/dist/'`

4. Creates React components: `App.tsx`, `HabitList.tsx`, `StreakView.tsx`, `WeeklyChart.tsx`

5. Components use `useMatrixDB('habits', ...)` and `useMatrixDB('entries', ...)` for data

6. Runs `pnpm install && pnpm build`

7. Verifies `dist/index.html` exists

8. App appears in dock, opens in iframe at `/files/apps/habit-tracker/dist/index.html`

9. Postgres tables `habit_tracker.habits` and `habit_tracker.entries` are auto-created by spec 050's boot registration

User from Telegram: "How's my exercise habit going this week?"
Agent queries: `MatrixOS.db.find('entries', { where: { habit_id: '...', date: { $gte: '2026-03-17' } } })` and responds with streak data.

Update `scanAppsDir` in `apps.ts` to read `manifest.entry` instead of hardcoding `index.html`. Skip framework apps where `build` is defined but the output directory does not exist (unbuilt apps). Write tests for entry resolution and unbuilt-app filtering.

### T3: AppManager entry field support
Update `register()` in `app-manager.ts` to use the `entry` field for path construction. Set status based on whether the build output exists. Write tests.

Create `packages/sdk/` workspace package. Implement `createMatrixOS()` with `db.*` (fetch-based), `getTheme()` (CSS custom properties), `generate()` / `openApp()` / `navigate()` (postMessage). Write unit tests for each method.

### T5: `@matrix-os/sdk` React hooks

Add `sudo` package and scoped sudoers rule for `/sbin/apk` to `Dockerfile.dev`. Verify the matrixos user can run `sudo apk add` but not other sudo commands.

### T8: E2E validation

## Non-Goals

- **Next.js / SSR apps**: requires process manager, port assignment, reverse proxy (future spec)
- **Hot reload / dev server in iframe**: user edits source, asks agent to rebuild (no HMR)
- **App store publishing of framework apps**: build artifacts are local only
- **Multi-framework templates beyond Vite + React**: add Vue/Svelte when requested
- **Build caching / incremental builds**: Vite is fast enough for now
- **Sandboxed build environments**: builds run in the user's container (trusted context)

## Dependencies

- Spec 050 (app data layer): Postgres backend, `MatrixOS.db.*` bridge, manifest `storage` field
- Existing: gateway file server, AppViewer iframe, OS bridge, app-manifest schema
- New: `@matrix-os/sdk` workspace package in `packages/sdk/`

## Success Metrics

- Agent scaffolds a Vite + React app from a natural language request
- `pnpm build` produces `dist/index.html` served correctly via `/files/`
- React app uses `useMatrixDB` to read/write Postgres data through the bridge
- Theme changes in the shell propagate to the React app via `useTheme()`
- User can say "rebuild my app" and the agent re-runs the build
- System deps (ffmpeg, etc.) install successfully via `sudo apk add`
- Existing single-file HTML apps continue to work unchanged
- `bun run test` passes with all new + existing tests
