# 098 — Onboarding polish: auth redesign + clickable billing with plan preselect

Status: Draft (design)
Date: 2026-06-28
Owner: hamedmp

## Summary

Three cohesive onboarding improvements on the marketing site (`www`) plus one
preselect read in the shell app:

1. Bring the sign-in / sign-up **left side** up to landing-page visual quality
   (calm editorial panel; drop the clashing gradient/grid/glass + auto-rotating
   keynote carousel).
2. Make each **landing pricing plan row clickable** → `/sign-up?plan=<slug>`.
3. **Preselect** the chosen plan later in onboarding, persisted in Clerk
   **public metadata**, with landing labels/prices **locked to the Stripe
   billing-select source of truth** via a parity test.

Non-negotiable constraint (added during brainstorming): **the onboarding state
machine is authoritative**. Preselect is UI intent only and must never skip,
fake, or reorder a journey phase.

## Background: the onboarding state machine

`packages/platform/src/journey.ts :: deriveJourneyPhase` is the source of truth.
Phases:

```
account_required → plan_required → payment_settling
  → install_choices_required → provisioning → (provisioning_failed)
  → first_run → ready
```

Phases are derived **server-side from real billing entitlement + machine
status** — never from client hints. The plan-picker entrypoint `?plans=1` is
**generated by the state machine itself** (`plan_required.nextAction.url =
plansUrl(appOrigin)`), so reaching it is part of the flow, not a bypass.

Render chain at `?plans`: `OnboardingGate → BillingGate → Settings →
BillingSection → BillingPanel`. `BillingPanel` holds `selectedProfileSlug`
state, today hardcoded to Builder (`MATRIX_BILLING_SERVER_PROFILES[1]`).

## Canonical plan data (the "Stripe billing select")

`shell/src/lib/billing.ts :: MATRIX_BILLING_SERVER_PROFILES` is the source of
truth consumed by the billing picker:

| label   | planSlug         | featureSlug   | hetzner | monthly | annual | specs              |
|---------|------------------|---------------|---------|---------|--------|--------------------|
| Starter | `matrix_starter` | `server_cpx22`| CPX22   | $14     | $140   | 2 vCPU/4GB/80GB     |
| Builder | `matrix_builder` | `server_cpx32`| CPX32   | $19     | $190   | 4 vCPU/8GB/160GB    |
| Max     | `matrix_max`     | `server_cpx52`| CPX52   | $49     | $490   | 12 vCPU/24GB/480GB  |

The landing currently duplicates these values inline in `LandingBilling.tsx`
(they happen to match today). We keep a www-local copy but **guard parity with a
cross-package test** so drift fails CI.

URL slugs (friendly, used in `?plan=`): `starter` | `builder` | `max`.

## Components & changes

### Part 1 — Auth left side (www, visual only)

- **Rewrite `www/src/components/auth/FeatureShowcase.tsx`** → still editorial
  panel. Remove: auto-advance carousel, `setInterval`, ember pills, animated
  progress segments, mono terminal visuals. Keep: `heading`/`subheading` props
  (sign-in vs sign-up differ). New layout: wordmark → Instrument Serif display
  headline (`fonts.display`) → muted sub → a clean 3-item proof list with small
  forest icons and hairline dividers (mirrors landing `valueProps`) → one quiet
  supporting element. Mobile: compact checklist (unchanged intent).
- **Rework `www/src/components/auth/AuthLayout.tsx`** → drop gradient + grid
  overlay + glass card. Cream page background (`#EEEEE2`); wrap the Clerk form in
  a landing-grade card (`#FCFCF8`, `rounded-2xl`, soft `cardShadow` from
  `landing/theme.ts`). Same two-column-on-`lg` / stacked-on-mobile structure.
- **Minor `clerkAppearance.ts`** radius/typography tweak so the form sits
  cleanly in the new card. No behavioral change.

No new state, fetches, or persistence. Purely presentational.

### Part 2 — Clickable landing billing (www)

- **`www/src/components/landing/LandingBilling.tsx`**: each plan row becomes an
  `<a href="/sign-up?plan={urlSlug}">` with hover affordance (cursor, subtle
  lift, trailing arrow) and telemetry `marketing_billing_plan_clicked`
  (`plan`, `location: pricing_section`). The existing "Choose a plan" button
  stays (→ `/sign-up`, no preselect).
- Plan data sourced from the new `www/src/lib/billing-plans.ts` (below) instead
  of the inline array.

### Part 3 — Preselect via Clerk public metadata (www → shell)

- **New `www/src/lib/billing-plans.ts`** — canonical www-local map:
  `urlSlug ↔ planSlug ↔ featureSlug ↔ label ↔ monthly/annual/specs`, plus
  `parsePlanUrlSlug(value): planSlug | null` (validation) and the display list
  for `LandingBilling`.
- **`www/src/app/sign-up/[[...sign-up]]/page.tsx`** (server component): read
  `searchParams.plan`; if it parses to a valid plan, set the Clerk
  `forceRedirectUrl`/`fallbackRedirectUrl` to `/welcome?plan=<urlSlug>` instead
  of the app URL. If absent/invalid, behavior is unchanged.
- **New route `www/src/app/welcome/route.ts`** (GET):
  - `auth()` → `userId`. If none, `redirect(appUrl)` (do not error).
  - `parsePlanUrlSlug(plan)`; if valid,
    `clerkClient().users.updateUserMetadata(userId, { publicMetadata: { selectedPlan: planSlug } })`.
    Wrap in try/catch: on failure, `console.error` (no provider/PII leak) and
    continue — a metadata write must never block onboarding.
  - `redirect(getMarketingAuthRedirectUrl())` → app **root** (normal boot
    sequence; the journey decides the phase).
- **`shell/src/components/settings/sections/BillingPanel.tsx`**: initialize
  `selectedProfileSlug` from `useUser().user?.publicMetadata?.selectedPlan`
  (mapped planSlug → featureSlug), falling back to today's Builder default when
  absent/unknown. Single-line-of-intent change; no journey coupling.

### Part 4 — Label/Stripe parity guard

- **New `tests/www/landing-billing-parity.test.ts`**: import
  `MATRIX_BILLING_SERVER_PROFILES` (shell) and the www `billing-plans` list;
  assert per planSlug that label, `monthlyPriceUsd`, `annualPriceUsd`, `vcpus`,
  `memoryGb`, `diskGb`, and `hetznerType` match. Locks landing == billing-select.

## Data flow

```
Landing pricing row click
  → /sign-up?plan=builder            (www, marketing site)
  → Clerk sign-up completes
  → forceRedirectUrl=/welcome?plan=builder
  → /welcome (server): write publicMetadata.selectedPlan=matrix_builder
  → redirect to app root /            (app.matrix-os.com)
  → BootSequence reads GET /api/journey → phase=plan_required (real entitlement)
  → user clicks "View plans" → ?plans=1  (URL the state machine produced)
  → BillingPanel: selectedProfileSlug ← publicMetadata.selectedPlan → Builder card preselected
  → Stripe checkout creates entitlement → journey advances normally
```

## Security architecture

- `/welcome` is **authenticated** (`auth()` userId required). Anonymous hits
  redirect to the app; they never write metadata.
- `plan` is **validated against an allowlist** (`parsePlanUrlSlug`) before any
  use. No raw value reaches `updateUserMetadata` or a URL.
- `publicMetadata.selectedPlan` is non-sensitive UX intent (one of three known
  slugs). It grants no entitlement; access control remains the journey +
  Stripe entitlement.
- No provider names / raw errors returned to the client; `/welcome` errors are
  logged server-side and the user is redirected with a generic outcome.

## Integration wiring

- Metadata is written **only** in www `/welcome` (www already has the Clerk
  backend wired via `@clerk/nextjs/server`); the shell only **reads** it. Avoids
  new backend plumbing in the shell.
- `BillingPanel` preselect is additive: it changes a default, not the picker’s
  behavior, validation, or checkout call.

## Failure modes

- **Clerk write fails / times out** → caught, logged, onboarding proceeds with
  the Builder default. No hang (server `redirect` always runs).
- **Invalid/absent `plan`** → ignored; today’s behavior.
- **publicMetadata has a stale/unknown slug** → `BillingPanel` falls back to the
  Builder default.
- **User abandons after `/welcome`** → intent persists in public metadata;
  returning later still preselects. The journey is unaffected.

## Resource management

- No new maps/sets, timers, temp files, or long-lived handles.
- Part 1 **removes** a `setInterval` (carousel), net reduction.

## Deferred / out of scope

- No shared `@matrix-os/billing-plans` package (parity test chosen over
  refactor). Revisit if a third consumer appears.
- No change to `deriveJourneyPhase`, checkout, or provisioning.
- No interval/region preselect — only the server profile (plan tier).

## Invariants

- **Source of truth**: journey phase = `deriveJourneyPhase` (entitlement +
  machine status). Plan labels/prices = `MATRIX_BILLING_SERVER_PROFILES`
  (shell), guarded by parity test.
- **Preselect is UI-only**: `publicMetadata.selectedPlan` defaults the picker;
  it never advances or skips a journey phase.
- **Auth source of truth**: `/welcome` requires Clerk `auth()`; anonymous never
  writes.
- **Acceptable orphan state**: metadata written but user abandons → harmless
  stale intent, overwritten on next plan choice.
- **Deferred scope**: shared package, interval/region preselect, journey edits.
