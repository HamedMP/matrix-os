# Implementation Plan: File Sync

**Branch**: `066-file-sync` | **Date**: 2026-04-13 | **Spec**: `specs/066-file-sync/spec.md`
**Input**: Feature specification from `/specs/066-file-sync/spec.md`

## Summary

Bidirectional file sync for Matrix OS ‚Äî Google Drive/Dropbox-style sync where every device is an equal peer. R2 is storage and transport; the gateway coordinates metadata, auth, and notifications but never touches file content. Clients upload/download directly to R2 via presigned URLs. Conflict resolution via 3-way merge (text) or conflict copies (binary). The system spans five subsystems: sync engine core (gateway + R2), CLI + local daemon, sharing/collaboration, remote access (SSH), and a Mac menu bar app.

**Key architectural decision**: R2 does not support `If-Match` on PutObject, so the spec's ETag-based optimistic concurrency on manifest writes must be implemented at the gateway level, not R2. The gateway becomes the sole writer of manifests, using Postgres-backed versioning to prevent concurrent updates.

## Technical Context

**Language/Version**: TypeScript 5.5+ strict, ES modules (gateway + sync client). Swift/SwiftUI (menu bar app).
**Primary Dependencies**: `@aws-sdk/client-s3` + `@aws-sdk/s3-request-presigner` (R2), Hono (gateway API), `citty` (CLI framework), chokidar v4 (file watching), `node-diff3` (3-way merge), Zod 4 (validation), Kysely (Postgres sharing table), `pino` (daemon logging)
**Storage**: Cloudflare R2 (file content + manifest), Postgres/Kysely (sharing permissions, sync metadata)
**Testing**: Vitest (unit + integration), TDD workflow
**Target Platform**: Node.js 24+ server (gateway), Node.js 24+ CLI (daemon), macOS/Linux (client), macOS (menu bar app)
**Project Type**: Monorepo ‚Äî gateway extensions + new `packages/sync-client/` package
**Performance Goals**: Sub-second sync notification latency for connected peers, <5s end-to-end for single file sync, support 100 concurrent presigned URL batch requests
**Constraints**: Manifest under 10K file entries, R2 presigned URLs max 7-day expiry (use 15 min), gateway never proxies file content, max 100 presigned URLs per batch request
**Scale/Scope**: Initial: 10-50 concurrent peers per user, 10K files per home directory, 100MB max file size before multipart

## Security Architecture

1. **Gateway auth**: Existing `authMiddleware` validates Bearer JWT on all `/api/sync/*` routes
2. **Path validation**: `resolveWithinPrefix(userId, path)` on every file path in presign/commit requests ‚Äî prevents path traversal
3. **Scoped R2 tokens**: Per-user R2 API tokens scoped to `matrixos-sync/{userId}/*`. Share tokens scoped to shared prefix only.
4. **Body limits**: `bodyLimit({ maxSize: 65536 })` on all mutating sync endpoints (metadata only, no file content)
5. **Timeouts**: `AbortSignal.timeout(10_000)` on all R2 API calls, `AbortSignal.timeout(30_000)` on presigned URL generation for large batches
6. **Rate limiting**: Per-user rate limits on presign requests (100 req/min) to prevent URL farming
7. **No error leakage**: R2 errors logged server-side, generic messages to client

### Phase 6 (OAuth) Architecture

The OAuth flow lives on the **platform service** (`packages/platform/`, port 9000), NOT the per-user gateway. Platform owns Clerk identity; gateways are single-tenant and trust JWTs the platform issues.

```
CLI (sync-client)        Platform (Clerk-aware)         Gateway (per-user)
     |                          |                              |
     |-- POST device/code ----->|                              |
     |<-- {deviceCode,userCode,verificationUri} ---|           |
     |-- (open browser) -------------------------->|           |
     |                          |                              |
     |                          | (user signs into Clerk,      |
     |                          |  POSTs userCode to /approve) |
     |                          |                              |
     |-- POST device/token ---->| (poll every 5s)              |
     |<-- {accessToken (JWT)} --|                              |
     |                          |                              |
     |-- HTTPS /api/sync/... -- with Bearer JWT --------------->|
     |                          |                              |--> validateSyncJwt()
     |                          |                              |    (verify signature,
     |                          |                              |     handle, exp, iss)
```

JWT claims: `{sub: clerkUserId, handle, gateway_url, iat, exp, iss: "matrix-os-platform"}`. Signed HS256 in dev (shared `PLATFORM_JWT_SECRET`), prep for RS256 in prod.
