# Upgrade Guide: 2026-04-02 Release

Covers all changes since the last deployment. This document is for the VPS coding agent to execute the upgrade on production.

## Summary of Changes

### Breaking Changes
1. **Image generation provider: fal.ai -> Gemini** -- `FAL_API_KEY` renamed to `GEMINI_API_KEY` across all packages
2. **`.next` cache volumes removed** from Docker Compose -- no longer persisted between container restarts
3. **Dashboard link changed** from `https://{handle}.matrix-os.com` to `https://app.matrix-os.com`
4. **PostgreSQL added to platform compose** -- shared instance, per-user database created on provision

### New Features
1. **Session-based routing** (`app.matrix-os.com`) -- Clerk JWT resolves user to container, no handle in URL
2. **BYOK (Bring Your Own Key)** -- users can supply their own Anthropic API key via `config.json`
3. **Setup screen** -- new first-run onboarding screen when no API key is configured
4. **API key validation endpoint** (`POST /api/settings/api-key`, `GET /api/settings/api-key/status`)
5. **Launcher** -- MissionControl renamed to "Launcher", removed task board / cron UI (simplified)
6. **Profile page redesign** -- hero section, stats strip, animated loading, empty states
7. **App tiles enlarged** -- icons 24x24 -> 96x96 in launcher, app icon images 115% bleed
8. **Command palette icons** -- shows app icons inline
9. **Canvas mode: focus on open** -- clicking a dock/app icon pans canvas to center on that window
10. **Games theme refresh** -- all 7 games use CSS custom properties for theming consistency
11. **Icon style updated** -- "neo-classic" dark matte design replaces macOS-style 3D icons

### Removed
- `home/apps/browser/` -- browser app removed
- `home/apps/code-editor/` -- code editor app removed
- `home/apps/vscode/` -- VS Code app removed
- `home/modules/hello-world/` -- hello-world module removed
- `home/plugins/hello-world/` -- hello-world plugin removed

### Internal Changes
- `createImageClient()` now uses Gemini API (`generativelanguage.googleapis.com`)
- `ImageResult` no longer has `url` field (only `localPath`)
- `GenerateOptions`: `size` -> `aspectRatio` + `imageSize`, `downloadFn` removed
- IPC `generate_image` tool: models changed to `gemini-2.5-flash-image` / `gemini-3.1-flash-image-preview` / `gemini-3-pro-image-preview`
- Dispatcher reads `config.json` for BYOK key, temporarily swaps `process.env.ANTHROPIC_API_KEY` during dispatch
- `getContainerByClerkId()` added to platform DB
- `focusOnWindow()` added to canvas transform store
- `focusOrOpen()` in Desktop -- focuses existing window instead of opening duplicate

## Step 1: Pull Latest Code (VPS)

```bash
cd /root/matrix-os   # or wherever the repo lives
git pull origin main
```

## Step 2: Update Environment Variables

### On VPS `.env` file:

```bash
# Replace FAL_API_KEY with GEMINI_API_KEY
sed -i 's/^FAL_API_KEY=.*/GEMINI_API_KEY=your-gemini-key-here/' .env

# If FAL_API_KEY doesn't exist as a line, just add GEMINI_API_KEY:
grep -q 'GEMINI_API_KEY' .env || echo 'GEMINI_API_KEY=your-gemini-key-here' >> .env

# Remove old FAL_API_KEY line if still present
sed -i '/^FAL_API_KEY=/d' .env
```

### On Vercel (www dashboard):

No env changes needed. The dashboard page now links to `app.matrix-os.com` instead of `{handle}.matrix-os.com` -- this is a code change, not an env change. Vercel auto-deploys from main.

## Step 3: Verify DNS / Cloudflare

The new `app.matrix-os.com` route needs to reach the platform service. If you have a wildcard CNAME (`*.matrix-os.com -> tunnel`), it already works. Verify:

```bash
# From VPS or any machine:
curl -sI https://app.matrix-os.com/health
# Should reach platform:9000 and return {"status":"ok"} (or redirect to login)
```

If the wildcard doesn't cover `app`, add a CNAME in Cloudflare DNS:
- Type: CNAME
- Name: `app`
- Target: `<tunnel-id>.cfargotunnel.com`
- Proxy: Yes

## Step 4: Rebuild and Restart Platform Services

```bash
# Rebuild platform + proxy (reads new GEMINI_API_KEY, starts PostgreSQL)
docker compose -f distro/docker-compose.platform.yml --env-file .env up -d --build
```

This starts/restarts the platform service with:
- **PostgreSQL** -- new shared instance, per-user database created automatically on provision
- New `app.matrix-os.com` session-based routing middleware
- `GEMINI_API_KEY` passed to user containers (replaces `FAL_API_KEY`)
- `getContainerByClerkId` DB function
- `POSTGRES_URL` env var pointing to the new PG instance
- Orchestrator now passes `DATABASE_URL=postgresql://matrixos:...@postgres:5432/matrixos_{handle}` to each user container

User containers keep running on the old image until upgraded. Existing containers won't have `DATABASE_URL` set until they are upgraded (step 6). New provisions automatically get a per-user database.

**For existing users**: the orchestrator now calls `createUserDatabase` in both `upgrade()` and `rollingRestart()`, not just `provision()`. This means existing users automatically get their per-user database created when their container is upgraded in step 6. No manual SQL needed.

## Step 5: Rebuild User Container Image

The user container image needs rebuilding for shell + gateway changes:

```bash
docker build \
  --build-arg NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY=pk_live_... \
  -t ghcr.io/hamedmp/matrix-os:latest \
  -f Dockerfile .
```

Changes baked into the new image:
- Shell: setup screen, launcher (simplified MissionControl), enlarged app tiles, command palette icons, canvas focus, profile redesign
- Gateway: Gemini image client, BYOK dispatcher, API key validation endpoints, onboarding module
- Kernel: Gemini image gen IPC tool
- Home template: removed browser/code-editor/vscode apps, removed hello-world module/plugin, updated games, new icon style, updated desktop.json

## Step 6: Upgrade Existing User Containers

Each running user container needs to be recreated with the new image. Use the upgrade API:

```bash
# List all containers
curl -s http://localhost:9000/containers \
  -H "Authorization: Bearer $PLATFORM_SECRET" | jq '.[] | .handle'

# Upgrade each one (stops old, creates new with same ports/volume)
for handle in $(curl -s http://localhost:9000/containers \
  -H "Authorization: Bearer $PLATFORM_SECRET" | jq -r '.[].handle'); do
  echo "Upgrading $handle..."
  curl -X POST "http://localhost:9000/containers/$handle/upgrade" \
    -H "Authorization: Bearer $PLATFORM_SECRET"
  echo ""
done
```

Or upgrade one at a time for safety:
```bash
curl -X POST http://localhost:9000/containers/alice/upgrade \
  -H "Authorization: Bearer $PLATFORM_SECRET"
```

## Step 7: Clean Up Old Volumes (Optional)

The `dev-next-cache` and `dev-next-cache-bob` volumes are no longer referenced. Safe to remove:

```bash
docker volume rm matrix-os_dev-next-cache matrix-os_dev-next-cache-bob 2>/dev/null || true
```

## Rollback Plan

If something goes wrong:

### Revert platform:
```bash
git checkout HEAD~1
docker compose -f distro/docker-compose.platform.yml --env-file .env up -d --build
```

### Revert a user container:
```bash
# Tag old image before rebuilding (do this BEFORE step 5)
docker tag ghcr.io/hamedmp/matrix-os:latest ghcr.io/hamedmp/matrix-os:pre-upgrade

# Rollback:
docker tag ghcr.io/hamedmp/matrix-os:pre-upgrade ghcr.io/hamedmp/matrix-os:latest
curl -X POST http://localhost:9000/containers/alice/upgrade \
  -H "Authorization: Bearer $PLATFORM_SECRET"
```

### Restore env:
```bash
sed -i 's/^GEMINI_API_KEY=.*/FAL_API_KEY=your-old-fal-key/' .env
```

## Notes for the Coding Agent

- **Do NOT `docker compose down -v`** -- volumes hold user data and node_modules
- **Do NOT push without testing** -- verify each step works before proceeding
- The `dev-next-cache` volumes only exist in `docker-compose.dev.yml` (local dev). On VPS using `docker-compose.platform.yml`, these volumes likely don't exist -- that's fine
- The `app.matrix-os.com` session routing depends on Clerk cookies being scoped to `.matrix-os.com`. If Clerk cookie domain isn't configured, users will get redirected to login in a loop. In that case, users can still use `{handle}.matrix-os.com` which doesn't require auth
- New user containers provisioned after the image rebuild automatically get the new image. Only existing containers need the upgrade API call
- The BYOK feature stores the API key in `~/system/config.json` under `kernel.anthropicApiKey`. The dispatcher temporarily swaps `process.env.ANTHROPIC_API_KEY` with this key during each dispatch, then restores it
- Icon regeneration will use the new Gemini model and "neo-classic" style. Existing icons are not auto-regenerated -- users can trigger via the desktop settings or the API
